# ZeroPress Theme Widget Runtime Contract

This document describes the current widget rendering contract used by the ZeroPress `v0.6` theme runtime.

It defines the responsibility boundary between:

- `preview-data.json`
- `@zeropress/build-core`
- theme templates

## Summary

Current model:

1. `preview-data.json` stores canonical widget configuration.
2. `@zeropress/build-core` resolves widget configuration into a theme-facing widget model.
3. theme templates render the resolved widget model.

Important rule:

- themes should render resolved widget fields
- themes should not interpret raw widget `settings`

## Source Of Truth

Canonical source data remains:

- `preview-data.json`
- `content.posts`
- `content.categories`
- `content.tags`
- `widgets.<area>.items[*]`

Resolved widget output is build-time render context only.

It is not written back into `preview-data.json`.

## Responsibility Split

### `preview-data.json`

Stores:

- widget area definitions
- widget order
- widget type
- widget title
- widget settings
- canonical content collections

Does not store:

- theme-specific widget markup
- duplicated widget render caches
- per-theme transformed arrays

### `@zeropress/build-core`

Resolves:

- item arrays
- widget-specific flags
- normalized URLs
- normalized media paths
- HTML for rich widget content when required

Also drops widgets that have no renderable output.

There is no theme-facing `empty` flag. Empty widgets are omitted from `widgets.<area>.items[]`.

### Theme Templates

Render:

- markup structure
- conditional presentation
- per-widget branches
- repeated item lists

Themes should not:

- resolve related content
- derive archive buckets
- apply widget limits
- interpret widget settings logic

## Canonical Input Shape

Example source-of-truth widget configuration:

```json
{
  "widgets": {
    "sidebar": {
      "name": "Sidebar",
      "items": [
        {
          "type": "recent-posts",
          "title": "Recent Posts",
          "settings": {
            "limit": 5,
            "show_date": true
          }
        }
      ]
    }
  }
}
```

Themes should not assume this source shape is the render contract.

## Resolved Widget Area Shape

Current theme-facing widget area shape:

```json
{
  "widgets": {
    "sidebar": {
      "name": "Sidebar",
      "items": [
        {
          "id": "sidebar-1",
          "type": "recent-posts",
          "title": "Recent Posts",
          "show_date": true,
          "items": [
            {
              "title": "Example Post",
              "url": "/posts/example-post/",
              "published_at": "Apr 23, 2026",
              "published_at_iso": "2026-04-23T10:00:00Z"
            }
          ]
        }
      ]
    }
  }
}
```

Common resolved fields:

| Field | Meaning |
| --- | --- |
| `id` | Stable render identifier generated by build-core |
| `type` | Stable widget type string |
| `title` | Widget title, may be empty |

Resolved fields:

- `avatar_url`
- `bio_text`

## Escaping And HTML Rules

Current interpretation rules:

- scalar fields are escaped by default
- `html` or `_html` fields are trusted raw HTML
- `_url` fields are normalized URL values prepared by build-core

Examples:

- `widget.title` is escaped
- `widget.display_name` is escaped
- `widget.html` is raw trusted HTML
- `widget.avatar_url` is a prepared URL

## Recommended Theme Consumption

Preferred:

```html
{{#for widget in widgets.sidebar.items}}
  {{#if_eq widget.type "profile"}}
    <section class="widget-card">
      {{#if widget.title}}<h2>{{widget.title}}</h2>{{/if}}
      {{#if widget.avatar_url}}<img src="{{widget.avatar_url}}" alt="{{widget.display_name}}">{{/if}}
      {{#if widget.display_name}}<p>{{widget.display_name}}</p>{{/if}}
      {{#if widget.bio_text}}<p>{{widget.bio_text}}</p>{{/if}}
    </section>
  {{/if_eq}}
{{/for}}
```

Not recommended:

```html
{{#for widget in widgets.sidebar.items}}
  {{#if_eq widget.type "profile"}}
    <img src="{{widget.settings.avatar}}" alt="{{widget.settings.display_name}}">
  {{/if_eq}}
{{/for}}
```

The current contract expects build-core to do configuration interpretation before the theme render step.

## Search Widget Note

The search widget provides render-ready UI fields, but search behavior still depends on theme JavaScript and the static search contract.

ZeroPress native builds can emit `/_zeropress/search.js`, a dependency-free ESM search adapter with a small Pagefind-like API. Search widgets should render the form and results container; the theme script can import the adapter and wire the UI.

Sites that use Pagefind can keep the same import path by running Pagefind into `/_zeropress/pagefind/` and replacing `/_zeropress/search.js` with `/_zeropress/search_pagefind.js` after the post-build Pagefind step.

## Empty Widget Behavior

Current behavior:

- empty widgets are omitted by build-core
- themes normally loop `widgets.<area>.items[]` directly
- themes do not need to filter by `empty`

## Recommended Author Workflow

1. Define widget configuration in `preview-data.json`.
2. Let build-core resolve widget render data.
3. Use `if`, `if_eq`, `for`, and partials to render markup in the theme.
4. Avoid reading raw widget `settings` in template code.
